🤖 RAG & AI Concepts - Beginner Guide

Last updated: June 2025

🎯 The Big Picture - Why RAG?

Problem: LLMs (like ChatGPT) only know what they were trained on. They can't answer questions about YOUR company data, new documents, or private info.

Solution: RAG = Retrieval Augmented Generation

Instead of retraining the whole model, you retrieve relevant info from your data and feed it to the LLM along with the question.

🍕 Simple Analogy:
LLM = A smart chef who knows all recipes
RAG = Giving that chef YOUR menu card before asking "What should I cook tonight?"
Without RAG → Chef gives generic answer
With RAG → Chef answers based on YOUR specific menu

✂️ 1. Chunking

What: Breaking large documents into smaller pieces (chunks)

Why: LLMs have token limits. You can't send a 500-page PDF at once. Smaller chunks = more precise retrieval.

Example:

Original document (1000 words about Azure VMs):

→ Chunk 1: "Azure VMs are virtual machines in the cloud..." (200 words)

→ Chunk 2: "VM sizes include B-series for dev/test..." (200 words)

→ Chunk 3: "To connect to a VM, use RDP or SSH..." (200 words)

Common Chunking Strategies:

StrategyHowWhen to Use
Fixed sizeEvery 500 charactersSimple, general docs
Sentence-basedSplit by sentencesShort docs, FAQs
Paragraph-basedSplit by paragraphsBlog posts, articles
OverlapChunks share some textAvoid losing context at edges
Overlap Example (chunk size=100, overlap=20):
Chunk 1: words 1-100
Chunk 2: words 80-180 (overlaps 20 words with chunk 1)
Chunk 3: words 160-260

🔢 2. Embedding

What: Converting text into numbers (vectors) that capture the meaning

Why: Computers can't understand "Azure VM" but they can compare numbers like [0.23, 0.87, 0.12...]

Key Idea: Similar meanings → similar numbers (close in space)

Example:

"dog" → [0.9, 0.1, 0.8, 0.2]

"puppy" → [0.85, 0.15, 0.75, 0.25] ← very close to "dog"!

"car" → [0.1, 0.9, 0.2, 0.7] ← very different from "dog"

🗄️ 3. Vector Database

What: A special database designed to store and search vectors (embeddings)

Why: Regular databases (SQL) search by exact match. Vector DBs search by similarity (closest meaning).

Regular DB vs Vector DB:

SQL: SELECT * FROM docs WHERE title = 'Azure VM' → exact match only

Vector DB: "Tell me about virtual machines in cloud" → finds "Azure VM" docs even without exact words!

📥 4. Data Ingestion Pipeline

What: The process of getting your raw data INTO the vector database (ready for search)

Steps: Load docs → Clean → Chunk → Embed → Store in Vector DB

📄 PDF, Word, Web pages, CSVs
↓ Load (read files)
🧹 Clean (remove junk, headers, footers)
↓ Chunk (split into pieces)
✂️ Chunks with metadata (source, page number)
↓ Embed (convert to vectors)
🔢 Vectors
↓ Store
🗄️ Vector Database
Real Example - Ingesting company HR policy:
1. Load: Read HR_Policy.pdf (50 pages)
2. Clean: Remove headers/footers/page numbers
3. Chunk: Split into 200 chunks (overlap 50 chars)
4. Embed: Convert each chunk → vector using Amazon Titan
5. Store: Save all 200 vectors in OpenSearch with metadata (page#, section name)

🔍 5. Smart Search (Semantic Search)

What: Finding relevant chunks based on MEANING, not just keywords

How: Convert the user's question → vector → find closest vectors in DB

Example:

User asks: "How many vacation days do I get?"

Search finds chunk: "Employees are entitled to 15 days of paid time off annually"

→ No exact word match ("vacation" vs "paid time off") but meaning is similar!

Search Types:

TypeHowBest For
Keyword searchExact word matchKnown terms
Semantic searchMeaning-based (vectors)Natural questions
Hybrid searchBoth combinedBest results (recommended)

🤝 6. How It Interacts with the LLM

The Final Step: Take retrieved chunks + user question → send to LLM → get answer

What actually gets sent to the LLM:

SYSTEM: You are a helpful assistant. Answer based ONLY on the context below.

CONTEXT:
[Chunk 1]: "Employees get 15 days PTO per year..."
[Chunk 2]: "After 5 years, PTO increases to 20 days..."

USER QUESTION: "How many vacation days do I get?"

LLM ANSWER: "You get 15 days of PTO per year. After 5 years, it increases to 20 days."

Key Points:

  • Context window: LLMs have a limit (e.g., 4K, 8K, 128K tokens). Retrieved chunks must fit within this.
  • Grounding: LLM answers based on YOUR data, not its training data → reduces hallucination
  • Prompt template: You control how the LLM uses the context (e.g., "only answer from context")

🏗️ Complete RAG Architecture

OFFLINE (one-time setup):
Documents → Chunking → Embedding → Store in Vector DB

ONLINE (every user query):
User Question → Embed Question → Search Vector DB → Get Top 3-5 Chunks → Send (Question + Chunks) to LLM → Return Answer to User

💼 Interview Questions & Answers

Q: What is RAG?
A: RAG (Retrieval Augmented Generation) is a technique where you retrieve relevant documents from a knowledge base and provide them as context to an LLM, so it can generate accurate answers based on your specific data.
Q: Why use RAG instead of fine-tuning?
A: RAG is cheaper, faster to set up, data stays fresh (just update docs), no retraining needed. Fine-tuning is better for changing the model's style/behavior.
Q: What is chunking and why is it important?
A: Chunking is splitting documents into smaller pieces. It's important because: (1) LLMs have token limits, (2) smaller chunks give more precise search results, (3) reduces noise in context.
Q: What is an embedding?
A: An embedding is a numerical representation (vector) of text that captures its semantic meaning. Similar texts have similar vectors, enabling meaning-based search.
Q: Vector DB vs Regular DB?
A: Regular DBs use exact match (WHERE name='X'). Vector DBs use similarity search — they find the closest vectors by meaning, even without exact word matches.
Q: What is semantic search?
A: Searching by meaning rather than keywords. "vacation days" finds "paid time off" because the vectors are similar even though words are different.
Q: How do you reduce hallucination in RAG?
A: (1) Use "answer only from context" in prompt, (2) Retrieve more relevant chunks, (3) Add source citations, (4) Use lower temperature setting.
Q: What is a data ingestion pipeline?
A: The process of loading raw documents, cleaning them, splitting into chunks, generating embeddings, and storing them in a vector database for later retrieval.
